# H Memo KST 일별 백업 및 365일 보존 설계

- 작성일: 2026-07-13
- 기준 시간대: `Asia/Seoul`
- 상태: 사용자 승인 완료

## 1. 목적

H Memo의 Windows, macOS, 웹앱이 동일한 Firestore 백업을 사용하면서도 서버 읽기·쓰기·저장 공간 사용량을 줄인다. 사용자는 하루에 여러 번 백업할 수 있지만, 복원 기록에는 대한민국 날짜별로 가장 마지막에 완료된 백업 1개만 남는다.

오늘을 포함한 최근 365개의 한국 달력 날짜만 보관한다. 백업하지 않은 날짜에는 빈 문서나 자리 표시자를 만들지 않는다. 따라서 최대 365개의 일별 복원 지점을 제공하되 실제 저장 개수는 사용자가 백업한 날짜 수만큼만 존재한다.

## 2. 확정 요구사항

1. 모든 날짜 경계와 보존 기간은 `Asia/Seoul` 기준으로 계산한다.
2. 같은 KST 날짜에 여러 번 백업하면 가장 늦게 완료된 백업만 복원 목록에 표시한다.
3. 같은 KST 날짜에 백업 내용까지 같으면 Firestore 쓰기를 생략한다.
4. 내용이 달라지면 새 불변 스냅샷을 먼저 완전히 저장하고 활성화한 뒤 기존 같은 날짜 백업을 정리한다.
5. 오늘과 직전 364개 KST 날짜의 백업만 유지한다.
6. 백업하지 않은 날짜에는 Firestore 문서를 만들지 않는다.
7. 기존 스키마 v1·v2 백업은 계속 읽고 복원할 수 있다.
8. 기존 중복·만료 데이터는 새 백업 성공 후 점진적으로 정리한다.
9. 정리 실패는 새 백업 성공을 취소하지 않으며 다음 백업에서 다시 시도한다.
10. 복원 목록은 메타데이터만 조회하고, 선택한 백업의 메모 본문만 지연 로드한다.

## 3. 범위 밖 항목

- 실시간 양방향 동기화
- 여러 기기의 동시 편집 충돌 자동 병합
- 백업하지 않은 날짜의 빈 복원 지점 생성
- 365일을 넘는 유료 장기 보관 정책
- 메모 첨부 파일과 이미지의 중복 제거
- 기존 전체 Firestore 데이터를 한 번에 삭제하는 위험한 마이그레이션

## 4. 저장 전략

### 4.1 선저장 후정리

백업은 다음 순서로 처리한다.

1. 로컬 메모를 정규화하고 콘텐츠 해시를 계산한다.
2. 현재 활성 백업 메타데이터를 확인한다.
3. 활성 백업의 KST 날짜가 오늘과 같고 콘텐츠 해시도 같으면 저장을 생략한다.
4. 변경된 내용이면 임의 ID를 가진 새로운 불변 스냅샷을 `writing` 상태로 만든다.
5. 모든 메모 본문을 스냅샷 하위 컬렉션에 기록한다.
6. 스냅샷을 `complete`로 전환하고 서버 `savedAt`을 기록한다.
7. `backupState/current`가 새 스냅샷을 가리키도록 원자적으로 활성화한다.
8. 활성화가 끝난 뒤에만 같은 날짜의 이전 백업과 365일 범위를 벗어난 백업을 정리한다.

이 순서는 정리 도중 네트워크 연결이 끊기거나 권한 오류가 발생해도 최신 백업이 먼저 보호되게 한다.

### 4.2 동일 내용 판정

콘텐츠 해시는 실제 클라우드 백업 페이로드를 안정적인 키 순서와 메모 ID 순서로 직렬화한 뒤 SHA-256으로 계산한다. 클라우드 복원 결과에 포함되지 않는 일시적인 UI 상태나 전송 상태만 해시에서 제외한다.

다음 조건을 모두 만족하면 서버 쓰기를 생략한다.

- 활성 백업 상태가 `complete`이다.
- 활성 백업의 서버 `savedAt`을 KST로 변환한 날짜가 현재 KST 날짜와 같다.
- 활성 백업의 `contentHash`가 현재 페이로드 해시와 같다.

현재 KST 날짜는 클라이언트 시계로 사전 판정하지만, 실제로 저장된 백업의 날짜 분류는 Firestore 서버 `savedAt`을 기준으로 한다. 자정 부근에서 클라이언트 시계가 어긋나면 중복 쓰기가 한 번 발생할 수 있지만 데이터 손실은 발생하지 않는다. 생성된 중복은 후속 정리에서 제거한다.

## 5. Firestore 스키마 v3

### 5.1 스냅샷 메타데이터

경로:

```text
users/{uid}/backupSnapshots/{snapshotId}
```

필드:

```ts
type BackupSnapshotMetadataV3 = {
  schemaVersion: 3;
  userId: string;
  state: "writing" | "complete";
  memoCount: number;
  contentHash: string;
  previewText: string;
  clientCreatedAt: string;
  savedAt: FirestoreTimestamp | null;
};
```

- `savedAt`은 완료 시점의 Firestore 서버 타임스탬프이다.
- `previewText`는 목록에서 메모 본문을 읽지 않고도 내용을 식별하기 위한 짧은 문자열이다.
- 별도의 빈 일자 문서나 KST 날짜 인덱스 문서는 만들지 않는다.
- KST 날짜 키는 `savedAt`을 `Asia/Seoul`로 변환해 계산한다.

### 5.2 메모 본문

경로:

```text
users/{uid}/backupSnapshots/{snapshotId}/memosV3/{encodedMemoId}
```

각 문서는 기존 검증 가능한 메모 백업 페이로드를 저장한다. 메모 ID는 문서 ID로 안전하게 인코딩하며 스냅샷이 활성화된 뒤에는 수정하지 않는다.

### 5.3 활성 백업 포인터

경로:

```text
users/{uid}/backupState/current
```

활성 스냅샷 ID와 스키마 버전을 저장한다. 정리 작업은 이 문서가 가리키는 스냅샷을 어떤 경우에도 삭제하지 않는다.

## 6. KST 보존 정책

### 6.1 보존 범위

KST 오늘 날짜를 `D`라고 할 때 보존 시작일은 `D - 364일`이다.

- 시작일 이상 오늘 이하의 날짜만 사용자 복원 목록에 포함한다.
- 날짜별로 서버 `savedAt`이 가장 늦은 `complete` 스냅샷 1개만 선택한다.
- 백업이 없는 날짜는 목록에 나타나지 않는다.
- 미래 날짜나 비정상 타임스탬프는 복원 목록에서 숨기되 활성 스냅샷이면 삭제하지 않는다.

달력 날짜 계산은 고정된 24시간 밀리초 차감이 아니라 `Asia/Seoul` 날짜 구성요소를 사용한다. 이 원칙으로 월말, 연말, 윤년 경계를 일관되게 처리한다.

### 6.2 점진적 정리 우선순위

성공한 백업마다 최대 400개의 Firestore 삭제 작업만 수행한다.

1. 현재 KST 날짜에 속한 비활성 중복 스냅샷
2. 최근 365일 내 날짜별 비활성 중복 스냅샷
3. 보존 시작일보다 오래된 비활성 스냅샷
4. 안전하게 분류 가능한 기존 v1·v2 비활성 스냅샷

하위 메모 문서를 먼저 삭제하고 하위 컬렉션이 비어 있는 것이 확인된 뒤 부모 메타데이터를 삭제한다. 작업 한도를 다 쓰면 부모 문서를 남기고 다음 성공한 백업에서 이어서 정리한다.

## 7. 기존 데이터 호환성

- v1·v2 스냅샷은 기존 리더로 계속 복원한다.
- 새 백업만 v3로 작성한다.
- 기존 백업의 날짜는 서버 `savedAt`을 우선 사용한다.
- 서버 시각이 없는 구형 데이터는 검증 가능한 `clientCreatedAt` 또는 기존 생성 시각을 보조 기준으로 사용한다.
- 날짜를 신뢰할 수 없는 기존 데이터는 자동 삭제하지 않고 복원 목록에서 별도 호환 항목으로 유지한다.
- 기존 데이터 정리는 사용자가 새 백업에 성공한 뒤에만 시작한다.

## 8. 조회 및 복원 흐름

### 8.1 메타데이터 전용 목록

기존의 전체 스냅샷 로드는 두 단계 API로 분리한다.

```ts
listBackupSnapshotSummaries(uid): Promise<BackupSnapshotSummary[]>
loadBackupSnapshot(uid, snapshotId): Promise<BackupPayload>
```

목록 API는 메타데이터만 읽고 다음 정보를 반환한다.

- 스냅샷 ID
- KST 백업 날짜
- 마지막 백업 시각
- 메모 수
- 미리보기
- 스키마 버전

목록 결과는 최근 365일 범위에서 KST 날짜별 최신 1개로 축약한다. 사용자가 특정 날짜를 선택할 때만 본문 API가 해당 스냅샷의 메모 문서를 읽는다.

### 8.2 복원 안전장치

복원은 다음 순서로 처리한다.

1. 사용자가 날짜별 백업을 선택한다.
2. 선택한 스냅샷 본문을 모두 읽고 스키마를 검증한다.
3. 현재 로컬 메모를 복원 안전 지점으로 저장한다.
4. 검증된 백업을 로컬 저장소에 적용한다.
5. 적용 도중 실패하면 기존 로컬 메모를 유지하거나 안전 지점으로 되돌린다.

복원 목록 조회 실패와 선택한 본문 로드 실패를 구분해 안내한다.

## 9. 사용자 인터페이스

복원 선택 화면에는 실제 백업이 있는 날짜만 최신순으로 표시한다.

각 항목에는 다음 정보를 보여준다.

- `YYYY-MM-DD` 형식의 KST 날짜
- KST 마지막 백업 시각
- 메모 개수
- 짧은 미리보기
- 복원 버튼

백업 결과 메시지는 다음 상태를 구분한다.

- 새 백업 완료
- 오늘 백업을 최신 내용으로 교체 완료
- 변경된 내용이 없어 저장 생략
- 백업 완료, 이전 기록 정리는 다음에 재시도
- 백업 실패

Windows, macOS, 웹앱은 같은 요약 모델과 공용 복원 선택 컴포넌트를 사용한다. 앱의 기존 `업데이트 내역`에는 2026-07-13 개선 사항으로 KST 일별 최신 백업과 365일 복원 기능을 추가한다.

## 10. 오류 처리

- 새 스냅샷 작성 실패: 활성 포인터와 기존 백업을 변경하지 않는다.
- 활성화 실패: 새 스냅샷을 복원 대상으로 노출하지 않고 기존 활성 백업을 유지한다.
- 정리 실패: 백업은 성공으로 유지하고 비치명적 정리 대기 상태를 반환한다.
- 목록 조회 실패: 로컬 메모와 서버 데이터를 변경하지 않는다.
- 본문 지연 로드 실패: 복원을 시작하지 않는다.
- 검증 실패: 로컬 저장소를 덮어쓰지 않는다.
- 권한 오류: 다른 사용자의 데이터 경로를 재시도하지 않는다.

## 11. 보안 규칙

Firestore 규칙은 다음 조건을 강제한다.

- 인증 UID와 경로 UID가 같은 사용자만 읽고 쓸 수 있다.
- v3 메타데이터 생성 시 허용 필드, 상태, 사용자 ID, 해시 형식을 검증한다.
- `writing`에서 `complete`로의 허용된 전환만 승인한다.
- 스냅샷 본문은 생성 후 수정하지 못한다.
- 본인 소유의 비활성 스냅샷과 그 하위 문서만 삭제할 수 있다.
- 현재 활성 포인터가 가리키는 백업은 클라이언트 정리 로직과 규칙 테스트에서 모두 보호한다.
- v1·v2 읽기 호환성은 유지한다.

다른 UID 접근, 잘못된 상태 전환, 활성 백업 삭제, 허용되지 않은 필드 추가를 규칙 테스트에 포함한다.

## 12. 코드 구성과 파일 크기

현재 500줄이 넘는 백업 모듈에 기능을 추가하지 않는다. 다음 책임을 500줄 미만의 모듈로 추출한다.

```text
packages/memo-sync/src/
  backup.ts                       # 공개 API 조립과 호환 계층
  backupFingerprint.ts            # 안정적 직렬화와 SHA-256
  backupKstDate.ts                # KST 날짜 키와 365일 범위
  backupRetention.ts              # 날짜별 선택과 정리 계획
  backupSnapshotSummary.ts        # 메타데이터 요약 파싱
  firestoreBackupCleanup.ts       # 제한된 Firestore 삭제 실행
```

백업 기록 UI도 공용 컴포넌트로 추출해 큰 Desktop/Web 앱 파일에 JSX를 더 쌓지 않는다. 구현 과정에서 수정하는 코드 파일이 500줄을 넘으면 해당 책임을 추가로 분리한다.

## 14. 완료 기준

1. 같은 KST 날짜에는 복원 가능한 최신 백업이 하나만 표시된다.
2. 같은 날 같은 내용의 반복 백업은 Firestore 쓰기를 만들지 않는다.
3. 같은 날 내용이 바뀌면 마지막 백업만 표시되고 이전 백업은 안전하게 정리된다.
4. 최근 365개 달력 날짜 밖의 비활성 백업은 점진적으로 삭제된다.
5. 백업하지 않은 날짜에는 문서가 생성되지 않는다.
6. v1·v2 백업을 계속 복원할 수 있다.
7. 복원 목록 조회만으로 메모 본문 문서를 읽지 않는다.
8. 정리 실패가 최신 활성 백업을 손상시키지 않는다.
9. 모든 플랫폼이 동일한 KST 날짜·보존 정책을 사용한다.
10. 수정·추가된 코드 파일은 책임별로 분리되어 500줄 미만을 유지한다.

## 15. 예상 사용량 개선

- 동일 내용 반복 백업의 쓰기 횟수를 0으로 줄인다.
- 하루 여러 백업의 장기 보관을 날짜별 1개로 제한한다.
- 최대 보존 범위를 최근 365개 달력 날짜로 제한한다.
- 복원 목록에서 본문을 읽지 않아 목록 조회 비용을 메타데이터 문서 수 수준으로 낮춘다.
- 정리를 작은 단위로 나눠 한 번의 백업에서 발생하는 삭제 부하를 제한한다.

추가적인 콘텐츠 주소 기반 중복 저장은 향후 첨부 파일 기능이나 대용량 본문이 도입될 때 검토한다. 현재 단계에서는 복잡도 대비 효과가 낮으므로 포함하지 않는다.
