# H Memo Firestore 백업 기록 커서 페이지네이션 계획

## 목표

- 서버 복원 진입 시 최근 365일의 백업 기록을 한 번에 모두 읽지 않습니다.
- Firestore에서 최신순으로 최대 10개만 화면에 전달합니다.
- 다음 페이지는 마지막 Firestore 문서 스냅샷을 커서로 이어 조회합니다.
- 이전 페이지는 메모리에 캐시한 결과를 사용해 Firestore를 다시 읽지 않습니다.
- v3 기록이 없는 기존 계정은 현재 전체 조회 호환 경로를 사용합니다.

## 데이터 계약

- 정렬: `savedAt desc`, 문서 ID `desc`
- 범위: 대한민국 날짜 기준 오늘부터 최근 365일
- 조회 크기: 화면 10개 + 다음 페이지 존재 확인용 1개
- 커서: Firestore `DocumentSnapshot`을 감싼 불투명 커서
- 반환: `{ summaries, nextCursor }`

## 구현

1. 백업 타입에 페이지 요청·결과·커서 타입을 추가합니다.
2. Firestore 드라이버에 KST 날짜 범위·정렬·limit·startAfter 쿼리를 추가합니다.
3. Firestore 게이트웨이에 `listBackupSummaryPage`를 구현합니다.
4. 공개 백업 API에 `listBackupSnapshotSummaryPage`를 추가하고 기존 게이트웨이 대체 경로를 유지합니다.
5. 데스크톱·웹 복원 상태를 페이지 캐시 구조로 바꾸고 다음·이전 핸들러를 연결합니다.
6. 공용 대화상자는 서버 페이지 상태를 표시하고 비동기 페이지 이동을 지원합니다.

## 안전장치

- 사용자 ID가 다른 문서는 결과에서 제외합니다.
- 유효하지 않거나 작성 중인 스냅샷은 기존 파서를 통해 제외합니다.
- 복원 확인과 안전 지점 생성은 기존 흐름을 유지합니다.
- 첫 페이지가 비었고 v3 결과가 없을 때만 기존 전체 조회로 대체합니다.

## 검증

- 25개 기록을 10·10·5개로 조회합니다.
- 첫 페이지에서 Firestore가 최대 11개 문서만 반환하는지 확인합니다.
- 다음 커서가 중복·누락 없이 이어지는지 확인합니다.
- 이전 페이지 이동에서 추가 서버 조회가 발생하지 않는지 앱 테스트로 확인합니다.
- 전체 테스트, Firestore 규칙 테스트, 타입 검사, 데스크톱·웹 빌드를 실행합니다.
- `npm run version:bump` 전후로 `npm run check:versions`를 실행합니다.
